## 二、苏格拉底追问（诘问挖本质找假设）

### 2.1 关于架构整体的追问

| 追问 | 本质 | 假设检验 |
|------|------|---------|
| **Q1: 为什么需要三层递进？** | 分而治之 + 粒度匹配 | 假设1：不同粒度的信息需要不同处理策略。若单一层处理，则无法兼顾SRAM级效率和全局语义。 |
| **Q2: 三层之间的信息传递是否闭环？** | 闭环检验 | 假设2：L1→L2→L3是前向流，但反馈从哪里来？目前只有跨尺度反馈，缺乏从L3/L4回L1的路径。**❌断层** |
| **Q3: 配置中心的假设是否成立？** | 无硬编码原则 | 假设3：所有参数都应可配置。但d_model在多处重复定义（L1/L2/L3/顶层），一处改则全部改。**❌冗余** |
| **Q4: 涌现机制的可验证性？** | 理论自洽 | 假设4：三元核心能涌现。但verify_theory.py中的验证是模拟的，未与真实Transformer行为对比。 |

### 3.2 L1 - 图书馆员类比

> **问题**：图书馆里有100万本书，如何让最常用的100本最容易找到？

**当前做法**（代码）：
1. 统计每本书被借阅的"专注度"（注意力熵）
2. 专注度高的书 → 放在前台（SRAM）
3. 专注度低的书 → 放回书架（HBM）

**大白话问题**：
- "专注度"怎么算？答：对一本书，问"谁在用它？用的时候有多专注？"
- 为什么有些书放在前台？答：不是馆员主观判断，而是根据使用模式统计
- 这种分类会变吗？答：会，每次借阅后重新计算

### 3.3 L2 - 专家组决策类比

> **问题**：有个复杂问题，需要找专家。但专家意见不一致怎么办？

**当前做法**（代码）：
1. 组建8人专家组，每次选2人发言（Top-K）
2. 选谁？→ 看谁过去的表现好（路由权重）
3. 但要允许新人表现，不能完全按历史（TTT在线学习）
4. 但不能让新人乱改，老人经验要保留（EWC）
5. 每次调整幅度要小，稳步前进（EMA）

**大白话问题**：
- 预训练是什么？→ 类似高考前的通识教育，打下基础
- EWC保护的"核心知识"是什么？→ 专家的基本功，不能丢
- TTT是什么？→ 入职后根据具体项目继续学习

### 3.4 L3 - 求解平衡点类比

> **问题**：两个互相拉扯的力，最终会停在哪儿？

**当前做法**（代码）：
1. 设一个初始位置
2. 计算两边的拉力
3. 移动一点，再算
4. 直到移动距离很小（收敛）

**大白话问题**：
- 可能永远不收敛？→ 对，所以加了断路器，到次数就停
- 可能震荡？→ 对，所以加了谱归一化，限制移动幅度
- 最终停在哪儿？→ 两个力的平衡点（不动点）

### 3.5 相变采样 - 水结冰类比

```python
class CrossScaleFeedback(nn.Module):
    def forward(self, macro_state, meso_state, micro_state, target_macro=None):
        # 计算与目标的误差（如果有）
        macro_error = 0.0
        if target_macro is not None:
            macro_error = F.mse_loss(macro_state, target_macro)
        
        # 基于误差调整反馈强度
        adjusted_alpha = self.macro_to_micro_scale * (1.0 + macro_error)
        adjusted_beta = self.macro_to_meso_scale * (1.0 + macro_error)
        
        # 误差驱动的反馈
        delta_router = -adjusted_alpha * macro_error * macro_state
        delta_cache = -adjusted_beta * macro_error * micro_state
        
        return delta_router, delta_cache, {
            "macro_error": macro_error.item(),
            "adjusted_alpha": adjusted_alpha,
            "adjusted_beta": adjusted_beta
        }
```

### 9.1 问题1: SRAM命中率0%

**症状**: 验证实验中SRAM命中率始终为0.0%

**根因分析**:
```
配置: d_model=128, sram_capacity=4096
Qwen: 每个token维度=3584
计算: 可容纳tokens = 4096 / 3584 = 1个 ❌ 严重不足
```

**解决方案**:
```yaml

l1_fusion:
  sram_capacity: 1048576   # 增加到1MB
  d_model: 3584            # 匹配Qwen模型
```

**结果**: SRAM命中率从0%提升到**100%** ✅

### 9.2 问题2: EWC预计算导致性能下降

**症状**: EWC预计算后生成时间翻倍 (19s → 37s)

**根因分析**:
```
EWC预计算前: 19.142s
EWC预计算后: 36.985s (+17.843s)

GPU内存分析:
- 初始状态: 12.74 GB
- L2路由器(无EWC): 13.57 GB (+0.83 GB)
- L2路由器(有EWC): 14.39 GB (+1.65 GB)
- EWC预计算后: 16.96 GB (+4.22 GB)

结论: EWC预计算分配约4GB GPU内存，导致模型参数offload到CPU
```

**解决方案**: 使用8-bit量化减少模型内存占用

```python
model = AutoModelForCausalLM.from_pretrained(
    model_path,
    load_in_8bit=True  # 启用8位压缩
)
```

**结果**: EWC预计算不再导致性能下降 ✅
